Detailed options for local development

Important

This is the entry point for the detailed, multi-option setup instructions I originally wrote when working out the requirements and my optimal setup and workflow for this project.

If you prefer a prescriptive, "decisions, not options" quick start guide, you can find that in the Local Dev Quick Start section.

  • Setup

Prerequisites

  • PHP and Composer installed locally
  • Node installed locally
  • Git installed locally
  • Sass installed globally on your machine
  • IDE of choice (using PhpStorm is documented throughout this site)

Note

The author of Comet Components is a Windows user and PhpStorm is her IDE of choice. While many of the instructions in this documentation and the convenience scripts provided are platform-agnostic (as she uses WSL which provides a Bash terminal, and many of the scripts are written in PHP or TypeScript), there are some things that developers using MacOS, Linux, and/or non-JetBrains IDEs will need to adapt for their own environments.

Tips

Windows-specific setup options and instructions are listed in the Windows setup deep-dive section below.

Setup

  1. Clone the repository from GitHub
  2. Install dependencies, refresh autoloading, and redo symlinks:
WSL (Bash)
npm run refresh:all
PowerShell
npm run refresh:all

See the CLI command quick reference for more options if you are returning to an already set up copy of the project and don't need to do a full refresh.

  1. Run the local web server and Storybook (at the same time i.e. two terminal windows) to see what you're working with!
WSL (Bash)
npm run test:server

npm run test:storybook
PowerShell
npm run test:server

npm run test:storybook

Tips

You can also run the project as a Laravel Herd site instead of running the basic local server, which enables you to use features like Herd's Dumps for debugging. See the Browser Testing documentation for more details.

  1. Optionally, run these docs locally:
WSL (Bash)
npm run docs
PowerShell
npm run docs

General notes and troubleshooting tips

About the PATH system environment variable

Making things like PHP and Composer available to Windows terminals generally involves adding their directory locations to your system path environment variable (often referred to as PATH in uppercase, even though Windows 11 calls it Path in the GUI).

Most installation methods handle this automatically, but there may be times that you need to add, delete, or change these paths manually, such as when:

  • you have multiple instances of PHP or Composer installed and your terminal aliases aren't using the one you want
  • you want to use an instance that doesn't automatically get added to the PATH, or it was optional during installation and you chose not to add it

You can find the GUI for these settings in Control Panel > System > Advanced System Settings > Environment Variables. (Or just search for "environment variables" in the Start menu.)

WSL (Bash)
readlink -f $(which php)

⚠️ Warning: WSL is not aware of the PATH settings unless you set up aliases to Windows executables that are. By default, it will look for instances you have installed in WSL itself.

To use the same versions as native Windows applications (such as Herd and PhpStorm) you can set up aliases to those. You can also set up aliases that go directly to specific executables, in which case it will also ignore PATH variables. See the WSL setup guide for more information.

For these reasons, if you switch between WSL and PowerShell you may get different results.

PowerShell
Get-Command php

⚠️ Warning: PowerShell in the PhpStorm terminal may pick up the version set as PhpStorm's CLI interpreter, rather than what you have set in your
PATH. There are two key PhpStorm settings that control this, which you can find in Settings > Tools > Terminal:

  • Add default project PHP interpreter to PATH
  • Shell integration.

If both of these are on, PowerShell will use the version set in PhpStorm and ignore anything else in your PATH when used in the PhpStorm terminal. Running a standalone PowerShell window will always use the version in your PATH.
